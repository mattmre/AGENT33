## Findings

### AEP-20260328-M01 — Operator Verification Runbook Scope Mismatch
- **severity**: Medium
- **component**: Operator runbooks / auth guidance
- **introduced by**: `#344`
- **files**:
  - `docs/operators/operator-verification-runbook.md:21,39,67,115`
  - `engine/src/agent33/api/routes/operator.py:45,96`
  - `engine/src/agent33/api/routes/processes.py:113`
  - `engine/src/agent33/api/routes/backups.py:53`
  - `engine/tests/test_operator_api.py:268`
  - `engine/tests/test_processes_api.py:79`
  - `engine/tests/test_backup_routes.py:96`
- **issue**: The runbook currently implies one `$TOKEN` can complete the full verification flow. Current shipped route scopes are split: read-only status and backup routes require `operator:read`, process inventory requires `processes:read`, and reset requires `operator:write` or broader elevation.
- **impact**: Operators following the runbook can fail partway through a live verification flow, or incorrectly assume reset can be run with a read-only token.
- **effort**: S
- **recommended minimal fix**: Add an explicit scope matrix or note to the runbook that separates the read-only verification token from the elevated reset token, and extend the doc test so the required scope strings are asserted.
- **acceptance criteria**:
  - The runbook states that process inventory requires `processes:read`.
  - The runbook states that reset requires `operator:write` or `admin`.
  - A doc regression test proves the scope guidance remains present.
- **dependencies**: None
- **blockers**: None

### AEP-20260328-L01 — Ingestion Artifact Planning Refs Are Not Repo-Root-Relative
- **severity**: Low
- **component**: Ingestion artifact helper / template
- **introduced by**: `#345`
- **files**:
  - `engine/src/agent33/improvement/ingestion_artifacts.py:169`
  - `engine/tests/test_ingestion_artifacts.py`
  - `docs/research/templates/INGESTION_TASK_TEMPLATE.md:16`
  - `docs/research/codex-autorunner-ingestion-task-2026-03-28.md:22`
- **issue**: `planning_refs` defaults are emitted as bare filenames, and the template/example repeat that convention. From the artifact location under `docs/research`, those values do not resolve to the repo-root planning files they are supposed to reference.
- **impact**: Operators or downstream tooling cannot reliably resolve the durable planning artifacts from the generated document alone.
- **effort**: S
- **recommended minimal fix**: Make `planning_refs` explicitly repo-root-relative, update the helper defaults, update the template/example wording, and add a regression test that verifies the documented refs resolve from repo root.
- **acceptance criteria**:
  - Helper defaults use one unambiguous reference convention.
  - Template and example match that convention.
  - A test proves each documented planning ref exists when resolved from repo root.
- **dependencies**: None
- **blockers**: None

### AEP-20260328-L02 — Repo-Ingestor Template Path Lacks Repo-Root Semantics
