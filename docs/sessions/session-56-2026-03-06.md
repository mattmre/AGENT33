## Phase 25 Stage 3 — WebSocket Workflow Status Streaming (PR #141)

### What was built

Real-time server-push for workflow execution state via a persistent WebSocket endpoint. Clients subscribe to a workflow run and receive typed events as execution progresses.

### New files

| File | Purpose |
|------|---------|
| `workflows/events.py` | `WorkflowEvent` base class + 8 concrete event types (started, node_entered, node_completed, node_failed, edge_traversed, workflow_completed, workflow_failed, heartbeat) |
| `workflows/ws_manager.py` | `WorkflowWSManager` — connection registry, fan-out broadcast, per-run subscription management |
| `api/routes/workflow_ws.py` | `/v1/workflows/{run_id}/ws` WebSocket endpoint with auth, subscription lifecycle |
| `docs/research/session56-stage3-research.md` | Design research and decision record |

### Modified files

| File | Change |
|------|--------|
| `main.py` | `ws_manager` added to app lifespan context; injected as dependency |
| `workflows/executor.py` | `event_sink` callback parameter; fires typed events at key execution points |
| `api/routes/workflows.py` | `ws_manager` wired in at route layer; passed to executor on each run |

### Key design decisions

- **Typed event model over raw dicts**: `WorkflowEvent` dataclasses enforced with a `type` discriminator field — consumers can pattern-match without defensive string checks.
- **Fan-out over per-connection queues**: `WorkflowWSManager` maintains a `dict[run_id, set[WebSocket]]`; events are broadcast to all subscribers of a run, not queued per connection. Simpler recovery path — reconnecting clients re-subscribe and get current state via a `sync` event.
- **Heartbeat cadence**: 30-second heartbeat prevents proxy timeouts on long-running workflows.
- **SSE fallback not implemented this session**: Noted as follow-up; WebSocket is the primary transport.

### Recovery note

The Phase 25 WebSocket implementation was present as **untracked files** in the `feat/session56-phase25-websocket-workflow` working tree from a prior failed session. These files were inspected, validated against the Phase 25 spec, committed, and pushed as PR #141 without re-implementing from scratch.

## Phase 28 Stage 3 — LLMGuard + Garak Real Security Adapters (PR #142)

### What was built

Replaced placeholder stub classes with fully-wired adapter implementations for two open-source LLM security libraries. Both adapters are optional-dependency-safe: they degrade gracefully to no-ops when the underlying library is not installed.

### Modified files

| File | Change |
|------|--------|
| `component_security/llm_security.py` | `LLMGuardAdapter` and `GarakAdapter` stubs replaced with real implementations; `_HAS_LLMGUARD` / `_HAS_GARAK` feature flags added |

### New files

| File | Purpose |
|------|---------|
| `tests/test_llm_security_adapters.py` | 31 tests (all fully mocked — no real LLMGuard/Garak install required in CI) |

### LLMGuardAdapter scanners

| Direction | Scanner |
|-----------|---------|
| Input | `PromptInjection`, `Toxicity`, `InvisibleText` |
| Output | `Sensitive`, `NoRefusal` |

### GarakAdapter probes

`promptinject`, `encoding`, `dan`, `leakreplay`

### Key design decisions

- **`_HAS_LLMGUARD` / `_HAS_GARAK` flags**: Guarded at import time with `try/except ImportError`. Adapter methods return `ScanResult.safe()` (pass-through) when the library is absent. This avoids adding hard dependencies to `pyproject.toml` — enterprise installs opt-in via `pip install agent33[security]`.
- **All 31 tests are mocked**: CI does not need the real libraries. Integration tests requiring live binaries are left for a dedicated security environment.
- **Scanner list is config-driven**: Passing a `scanners` list to `LLMGuardAdapter.__init__` overrides the defaults, making it easy to add or remove scanners per deployment without subclassing.

## Phase 43 Stage 2 — MCP Resources + Auth Scope Enforcement (PR #143)

### What was built

Two missing pieces of the MCP server: a resource registry with both static and URI-template resources, and an auth module that enforces OAuth2-style scopes on MCP routes. Also fixed a **critical wiring bug** where `MCPServiceBridge` was instantiated but never started in the app lifespan.

### New files

| File | Purpose |
|------|---------|
| `mcp_server/resources.py` | 4 static resources (`agent-registry`, `tool-catalog`, `policy-pack`, `schema-index`) + 3 URI-template resources (`agents/{id}`, `tools/{name}`, `workflows/{id}`) |
| `mcp_server/auth.py` | `require_scope()` dependency factory; `ScopeError` exception; scope hierarchy constants |
| `tests/test_mcp_resources.py` | Resource listing, template resolution, content fetch tests |
| `tests/test_mcp_resources_auth.py` | Scope enforcement tests — missing scope → 403, correct scope → pass |

### Modified files

| File | Change |
|------|--------|
| `mcp_server/bridge.py` | `register_resources()` method wired to resources module |
| `api/routes/mcp.py` | `require_scope` guards added to write/admin endpoints |
| `main.py` | **Bug fix**: `MCPServiceBridge` now started and stopped in `lifespan` context manager |

### Critical bug fixed

`MCPServiceBridge.__init__` was called during app startup but `bridge.start()` was never invoked in the lifespan context, meaning MCP transport handlers were registered but never activated. All MCP connections silently failed at the transport layer. Fixed by adding `await bridge.start()` / `await bridge.stop()` inside the async `lifespan` context manager in `main.py`.

### Key design decisions

- **Static + template resources in one registry**: `resources.py` exports a single `RESOURCE_REGISTRY` dict keyed by URI pattern. Template resources use `{param}` placeholders resolved at fetch time.
- **Scope hierarchy as constants**: `mcp_server/auth.py` defines `SCOPE_READ`, `SCOPE_WRITE`, `SCOPE_ADMIN` as string constants plus a `SCOPE_IMPLIES` dict for hierarchical enforcement (admin implies write implies read).

## Phase 38 Stage 2 — Token-Level LLM Streaming + Tool Call Reassembly (PR #144)

### What was built

End-to-end token streaming from LLM providers (OpenAI, Ollama) through the agent tool loop, with a `ToolCallAssembler` that reconstructs tool-call objects from streamed deltas before dispatch. This is the highest-test-count PR of the session (44 tests).

### New files

| File | Purpose |
|------|---------|
| `llm/stream_assembler.py` | `ToolCallAssembler` — stateful delta accumulator; emits complete `ToolCall` objects once the stream closes or a new tool call index begins |
| `llm/streaming.py` | Public re-export shim — `from llm.streaming import ToolCallAssembler, LLMStreamChunk` |
| `tests/test_stream_assembler.py` | 28 unit tests for assembler edge cases |
| `tests/test_streaming.py` | 16 integration tests for end-to-end streaming through agent loop |

### Modified files

| File | Change |
|------|--------|
| `llm/base.py` | Added `ToolCallDelta` dataclass; `LLMStreamChunk` gains optional `tool_call_delta` field |
| `llm/openai.py` | `delta.tool_calls` parsing — extracts `ToolCallDelta` from streaming response chunks |
| `llm/ollama.py` | Tool object emission — yields `ToolCallDelta` when Ollama streams tool objects |
| `agents/events.py` | `LLMTokenEvent` added for per-token agent loop events |
| `agents/tool_loop.py` | `stream_complete` accumulation path + fallback to non-streaming if provider doesn't support it |

### Key design decisions

- **`ToolCallAssembler` is stateless across calls but stateful within a stream**: A new assembler instance is created per LLM call. It accumulates `argument` string fragments indexed by tool call index and flushes on `close()`.
- **`llm/streaming.py` re-export shim**: Keeps the public API surface clean — callers import from `llm.streaming`, not from implementation modules. Consistent with the existing `llm/base.py` pattern.
- **Fallback path in `tool_loop.py`**: If the provider yields no `tool_call_delta` in any chunk (e.g. an older model that returns tool calls in the final chunk only), the loop falls back to the non-streaming tool-call extraction path. Zero regressions against existing tests.
- **`LLMTokenEvent` in `agents/events.py`**: Enables downstream consumers (WebSocket streaming, SSE) to forward individual tokens to clients without coupling the LLM layer to the transport layer.
