Why this sprint exists

The current product posture is too cryptic, too settings-heavy, and too expert-dependent

Current pain

  • Too much functionality hides behind menus, settings, and internal terminology.
  • Normal users do not get a clear first-run path or a clear explanation of what the system is doing.
  • Advanced capability leaks into default surfaces, making the product harder to trust and harder to start.

What changed in the plan

  • Layman usability is now a governing constraint, not an optional refinement.
  • Operator features are progressively disclosed and explicitly opt-in.
  • Bounded execution, evidence, replay, and reversible improvement are part of the product promise.
Key message We are fixing both the product surface and the underlying truth model
Strategic direction

AGENT-33 is a shared harness plus specialist-cadre ecosystem

Build for

  • Specialized agents over general-agent sprawl
  • Artifacts and evidence over ambient context
  • Bounded execution over loose continuity
  • Summary-first surfaces and human-legible judgment
  • Reversible improvement over silent self-mutation

Avoid

  • Cryptic subsystem menus as the main product identity
  • Telemetry, memory, or model scores becoming silent authority
  • Layman-facing prototypes that cannot explain themselves
  • Improvement workflows that outrun containment and rollback
Design principle UI consumes runtime truth; UI does not define runtime truth
Product promise

The first-run experience is now explicitly defined

Default layman path

  • See progress in plain language

UX guardrails

PM implication The starting experience is now part of scope, not left to implementation guesswork
Panel sweep 1

First expert audit: what the original final-plan draft was missing

Architecture flagged

  • Phase 3 was too early relative to replay and recovery semantics.
  • Layer 0 exit criteria were too vague.
  • Phase 4 / Phase 5 lacked a clear interface contract.
  • Gate 3 was under-decomposed.

UX flagged

  • Phase 3 was still too compressed.
  • Dashboard, inbox, evidence ladder, and replay explanation were under-specified.
  • The first visible slice was backend-correct but product-wrong.
  • The layman ask -> answer -> why flow was not prioritized enough.

Risk flagged

  • Cross-gate integration was weaker than individual gate text.
  • Proof-grade validation was undefined.
  • Rollback, blast-radius, and containment semantics were too weak.
  • Environment evidence promotion into shared memory was under-specified.
Why this matters The original final plan was directionally right but not review-hardened enough

Panel sweep 3

Final sweep: last-mile issues were all about first-run clarity

What the final sweep still found

  • First-run / setup / model-connection path was not explicit enough.
  • Non-approval statuses needed plain-language treatment.
  • Default first-run view needed to be explicitly layman.
  • Pre-slice prototype needed a strict exposure boundary.
  • Promotion review items needed to stay inactive until later phases.

What changed after the final sweep

  • Onboarding now includes install, model connection, role selection, and first task.
  • Layman view is the default first-run surface.
  • Layman-visible statuses now require user-language descriptions.
  • Prototype outputs are internal-only before Gate 4 completes.
  • Phase 4 implementation is explicitly blocked until the Phase 3 replay surface is locked.
Sweep 3 result No structural issues remained; only first-run and default-path quality needed tightening

Review-ready does not mean officially locked

Gate 2Compact task-state, checkpoint, frozen handoff, replay
Gate 4Memory lifecycle, provenance, precedence
Gate 3Context-budget and skill telemetry

Gate 6Improvement promotion, rollback, containment

Why lock is still blocked

  • The plan is strong enough for review and sprint planning.
  • The architecture is not declared frozen until Layer 0 proves the assumptions behind contracts, replay, memory, telemetry, environment evidence, and rollback.

Decision gate

  • GO - gates and integration evidence pass
  • REVISE - one or more interfaces need another bounded pass
  • STOP - a critical assumption fails
PM implication Start the sprint with validation discipline, not with an assumption that the architecture is already proven

Sprint recommendation

Recommended sprint objective: finish the backbone and lock the first visible experience

Track A - Layer 0 backbone

  • Complete the next Batch 1 validation items:
    • Gate 1 event / status / error semantics
    • Gate 2 checkpoint unit and frozen handoff bundle
    • Gate 4 memory classes and sharing boundaries
  • Preserve proof-grade artifacts, validators, and worked examples.

Track B - Product definition for Slice 1

  • Lock first-run onboarding and model-connection flow.
  • Lock layman progress, answer, and why-copy behavior.
  • Lock operator inbox shell with reserved forward-phase slots clearly inactive.
  • Keep the pre-slice prototype internal only until Gate 4 is complete.
What not to do Do not ship a public layman surface before replay + retrieval explanation depth is ready
First release promise

Slice 1 is designed to create user confidence fast

What the first visible slice must do

  • Give the user a simple ask -> answer -> why loop.
  • Show progress without subsystem leakage.
  • Explain approval waits and blocked states in user language.
  • Surface evidence as summary first, then allow drill-down.

What it must not do

  • Dump raw status tokens or internal model uncertainty codes on layman users.
  • Expose hidden operator complexity by default.
  • Turn telemetry into authority.
  • Offer environment or self-improvement controls as starter UX.
Success test A first-run user gets value without needing architecture documentation
Risk posture

The plan contains the high-risk areas instead of hiding them

Execution risk

  • Replay and checkpoint truth must be validated before stable surfaces ship.
  • Cross-gate integration needs human sign-off, not assertion.

User trust risk

  • Low-confidence and insufficient-evidence results are translated into user language.
  • Prototype outputs stay internal until explanation depth is real.

Improvement risk

  • Proposal-only lifecycle, staged rollout, blast-radius matrix, rollback ancestry.
  • No silent high-impact mutation.
PM implication Trust and containment are part of scope, not deferred cleanup
